# 第一版项目检查流程细节设计

## 目标边界

第一版目标是实现一个面向项目目录的检查 Agent。输入包括项目路径和用户首次文本说明，Agent 进入该目录后自行发现项目要求文件、代码、测试、状态文档和其他相关材料。第一版不接入联网搜索，所有判断先基于本地项目材料、PDF 内容、命令行执行结果、已有文档和用户提供的初始说明。

输出是一份检查报告。报告需要给出当前项目完成度的大致百分比，并列出已完成项目、未完成项目、部分完成项目、不确定项目，以及对应证据和下一步建议。

## 输入与输出

输入：

- `project_path`：被检查项目的根目录。
- `user_note`：用户首次输入的简短说明，用于补充项目背景、检查重点、已知提交方式或特殊约束。

Agent 不要求用户提前指定 PDF、代码目录、测试命令或提交物目录。第一步由 Agent 结合项目路径和用户文本说明自行扫描、分类和建立上下文。

输出：

- 项目检查报告。
- 报告内容包括完成度百分比、需求项状态、已完成内容、未完成内容、部分完成内容、不确定内容、证据来源、建议下一步。

## 第一版 Graph 节点

### 1. 抓取上下文

该节点负责进入项目目录并建立初始项目认知。它需要读取用户首次文本说明，扫描目录结构，初步识别哪些文件像项目要求，哪些是代码，哪些是测试，哪些是报告、设计文档、状态文档或提交材料。

PDF 要在第一步就进行初步读取。原因是 PDF 中可能包含项目要求、设计模板、评分标准或提交说明；如果不读取 PDF，Agent 无法判断该文件是否属于需求来源。第一步不要求记住 PDF 的所有细节，但必须能判断该 PDF 的大致类型，尤其要识别其中是否包含项目需求。

该节点的产物不是最终需求清单，而是上下文索引和材料分类结果。

### 2. 整理需求

该节点基于第一步标注出的需求来源进行结构化整理。主要来源包括项目要求 PDF、要求文档、README、课程说明、状态文档，以及代码仓库中对项目状态的补充说明。

整理结果是一个需求列表，不是额外的 checklist。列表需要让 Agent 清楚当前项目到底有哪些要求，包括功能要求、文档要求、测试要求、提交物要求和可能的评分点。表达上以“需求项”为主，每一项保留来源和简要说明。

### 3. 规划检查路径

该节点负责 planner 行为。它读取需求列表和项目上下文，规划后续检查应该怎么做，包括先检查哪些文件、是否需要运行命令、是否需要分给多个 Agent 分别整理、哪些部分应先确认、哪些部分可以延后。

该节点不再生成独立 checklist。它的职责是给出执行计划：接下来分几步查、每一步查什么、需要哪些工具或子 Agent、预期产出是什么。

### 4. 执行检查

该节点按 planner 的方案执行实际检查。检查范围包括代码、测试、文档、提交材料和命令行执行结果。它可以运行 PowerShell 命令，读取终端反馈，并把结果作为证据记录下来。

第一版只要求能执行命令行程序并拿到终端输出，不要求具备复杂的代码运行环境管理能力。命令失败时应记录失败原因、命令输出和对当前检查项的影响。

### 5. 打状态

该节点根据需求列表、执行证据和项目上下文，为每个需求项判断状态。状态先使用以下几类：

- `done`：证据显示已经完成。
- `partial`：有实现或文档，但存在缺口。
- `missing`：未发现对应实现或材料。
- `unknown`：现有证据不足，无法判断。

完成度百分比由这些状态汇总得出。第一版可以先使用粗略估算，不追求评分级精确，但报告中要说明估算依据。

### 6. 生成最终报告

该节点生成最终检查报告。报告写入文件是流程最后一步的写文档动作，不需要单独包装成一个专门的 `write_report` 工具。

报告应保持开发者视角，直接记录检查结论、证据和风险，不使用教程式或面向读者解释式表达。

## Reason Then Act 约束

每个关键节点都应遵循先 reason、再 act 的基本工作方式。模型在执行工具前应先形成简短判断：当前目标是什么、需要哪些信息、下一步动作为什么合理。随后再执行读取文件、解析 PDF、运行命令或写报告等动作。

该约束可以同时体现在系统提示词和节点流程设计中。第一版先通过提示词和节点职责约束实现，不强行设计复杂的内部思维存储结构。

## 核心状态草案

第一版状态可以先保持可用，不追求最终精确：

```python
ProjectCheckState = {
    "project_path": str,
    "user_note": str,
    "context_index": list,
    "requirement_sources": list,
    "requirements": list,
    "plan": list,
    "evidence": list,
    "findings": list,
    "completion_estimate": dict,
    "report_path": str,
}
```

字段含义：

- `project_path`：被检查项目根目录。
- `user_note`：用户首次输入的项目说明或检查重点。
- `context_index`：初步扫描出的项目材料索引和分类。
- `requirement_sources`：被判定为需求来源的文件或 PDF。
- `requirements`：结构化需求列表。
- `plan`：planner 生成的检查步骤。
- `evidence`：文件读取、PDF 解析、命令执行和测试输出形成的证据。
- `findings`：每个需求项的判断结果。
- `completion_estimate`：完成度百分比及估算依据。
- `report_path`：最终报告路径。

## 工具边界

第一版工具分为两类。

### 命令行类工具

命令行类工具负责本地文件和终端操作。`read_file`、`list_files`、`search_text` 等能力本质上都可以落到简单命令或文件系统操作上；重点能力是 `run_command`，尤其是能通过 PowerShell 执行命令并读取终端反馈。

命令行类工具需要记录命令、退出码、标准输出、标准错误和工作目录。

### PDF 与视觉识别类工具

PDF 内容提取是独立重点。普通命令行工具不一定能可靠处理 PDF，尤其当 PDF 中包含图片、扫描页、表格或截图时，需要单独的 PDF 提取能力，未来可能需要调用专门 Agent 或视觉识别工具。

第一版可以先实现基础文本提取；遇到图片型 PDF 时记录为能力缺口或转入后续阶段处理。

## 第一阶段不做的内容

- 不做联网搜索。
- 不做复杂 UI。
- 不强制支持用户中途交互。
- 不要求完整 OCR 或视觉理解能力。
- 不要求多 Agent 并行执行必须落地，只保留 planner 对多 Agent 拆分的设计位置。
- 不把报告写入封装成独立业务工具。
- 不处理没有实际可确认产物的要求，例如线下课程演示、现场答辩、向助教讲解代码原理等。
- 不生成独立的项目补充说明文档。

## 第二阶段保留设计

第二阶段需要加入用户交互能力。当 Agent 判断项目要求不完整、需求来源不清楚、PDF 无法可靠解析，或中期检查发现关键问题时，可以向用户提问。交互形式可以是弹窗或其他前端提示，由用户补充信息后继续流程。

第二阶段也可以强化 planner，让它把检查任务分给多个 Agent 分别整理，例如需求整理 Agent、代码检查 Agent、测试运行 Agent、文档检查 Agent、报告汇总 Agent。第一版只记录该方向，不作为必须实现项。

第二阶段需要处理没有实际可确认产物的要求。部分项目要求不是仓库内产物，例如线下演示、答辩准备、向助教讲解代码原理、现场展示运行效果等。这类要求第一阶段不纳入完成度判断；第二阶段可以单独拉出一份“非仓库产物要求清单”，由 Agent 标注其性质、确认方式和需要用户补充的信息。

第二阶段还需要支持项目补充说明文档。最终报告会记录检查结论，但当 Agent 在中期交互中获得额外信息时，例如用户说明视频会另行提交、某个材料不需要出现在仓库中，Agent 还应在报告同目录生成一份项目补充说明文档。该文档用于沉淀项目描述、外部提交方式、人工确认事项和对检查边界的补充说明，避免这些信息只留在一次对话或报告结论中。
